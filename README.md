# awto-wit

Python and Rust BLE clients for **WitMotion WT901BLE** 9‑axis IMUs, plus
register read/write tools for configuration.

> Tested device: **WT901BLE67** (BLE 5.0). The same protocol is used by other
> WitMotion BLE IMUs (e.g. WT9011DCL) — see notes below.
>
> Renamed from `test-auto-wit` on 2026-05-25.
>
> ⚠️ Initially generated by an AI assistant (GitHub Copilot / Claude).
> Register-protocol notes below are from direct experimentation against
> WT901BLE67 on that date; treat them as empirically verified for that
> firmware revision but not as official vendor documentation.

## Layout

- `py/` – Python (using [`bleak`](https://github.com/hbldh/bleak))
  - `wit_ble.py` — stream IMU data (accel/gyro/euler)
  - `wit_regs.py` — read configuration registers
  - `wit_write_lowpower.py` — write the auto-sleep timeout (register 0x25)
- `rs/` – Rust (using [`btleplug`](https://github.com/deviceplug/btleplug))
  - `src/main.rs` — stream IMU data

## Quickstart

### Python

```bash
cd py
python3 -m venv .venv
.venv/bin/pip install bleak
.venv/bin/python wit_ble.py [MAC] [N_SAMPLES]            # stream data
.venv/bin/python wit_regs.py [MAC]                       # dump config registers
.venv/bin/python wit_write_lowpower.py [SECONDS] [MAC]   # change sleep timeout
```

Default MAC is hard-coded; override on the command line.

### Rust

```bash
cd rs
cargo run --release
```

MAC is a constant in `src/main.rs`.

## BLE characteristics

| UUID | Direction | Purpose |
|------|-----------|---------|
| `0000ffe4-0000-1000-8000-00805f9a34fb` | Notify | IMU data frames + register-read responses |
| `0000ffe9-0000-1000-8000-00805f9a34fb` | Write (no response) | Config commands |

No pairing required for notifications. BlueZ ≥ 5.66 on Linux works fine.

## Data frame format (`0x55 0x61`)

20 bytes, little-endian `i16`s:

| Bytes | Field | Scale |
|------:|-------|-------|
| 0..1  | header `0x55 0x61` | – |
| 2..7  | ax, ay, az | `/32768 * 16` g |
| 8..13 | wx, wy, wz | `/32768 * 2000` °/s |
|14..19 | roll, pitch, yaw | `/32768 * 180` ° |

## Configuration protocol

All config commands are 5-byte packets to the write characteristic:

```
FF AA <reg> <data_lo> <data_hi>
```

### Reading registers

Request 8 consecutive registers starting at `<reg>`:

```
FF AA 27 <reg> 00
```

Response arrives on the notify characteristic as a 20-byte frame:

```
55 71 <reg> 00 <r0_lo r0_hi> <r1_lo r1_hi> ... <r7_lo r7_hi>
```

Note: this firmware uses **`55 71`** for register-read responses, not the
`55 5F` documented for some older WitMotion BLE units.

The data stream (`55 61` frames) continues during register reads — your parser
must accumulate notifications and dispatch by frame header.

### Writing registers

Registers are write-locked by default. Sequence is:

```
FF AA 69 88 B5      # unlock
FF AA <reg> <lo> <hi>   # set reg = hi:lo
FF AA 00 00 00      # save to flash (otherwise change is RAM-only)
```

The chip needs a brief moment between each step to process the opcode and
(for the save) to commit to flash. The exact timing isn't documented;
200ms between writes and 500ms after save have worked reliably in testing,
but should probably be tuned down for production use.

## Register map (WT901BLE67 firmware)

Verified empirically by reading and writing — not from official datasheets.

### Configuration (persist to flash via save)

| Reg | Name | Notes |
|-----|------|-------|
| `0x00` | SAVE | trigger flash save (write `0x0000`) |
| `0x01` | CALSW | calibration mode |
| `0x02` | RSW | output content mask |
| `0x03` | RRATE | output rate. `11` = 100 Hz (factory default) |
| `0x04` | BAUD | serial baud (not relevant for BLE) |
| `0x05`–`0x0A` | offsets | accel/gyro zero offsets |
| `0x12`–`0x16` | NAME | ASCII device name, e.g. "WT901BLE67" |
| `0x22` | SLEEP | one-shot: write `1` to sleep now |
| `0x23` | ORIENT | orientation hint |
| `0x24` | POWERONSEND | (purpose unclear; was 0 on test unit) |
| **`0x25`** | **LOWPOWER** | **seconds before sleep when not connected.** NOT a bool. Factory default `30`. Write `3600` for 1 hour, larger for longer. |
| `0x66`–`0x68` | MAC | MAC address in reverse byte order |
| `0x69` | UNLOCK | unlock register (write `0x88B5`) |

### Live measurements (read-only, change between reads)

| Reg | Notes |
|-----|-------|
| `0x08`–`0x0A` | gyro live offsets (calibration is dynamic) |
| `0x1C`–`0x1F` | unknown live data |
| `0x38`–`0x3F` | unknown live data |

## Battery / sleep behaviour

The WT901BLE67 advertises continuously when powered on, then stops
advertising after `LOWPOWER` seconds (factory default 30s) when not
connected — even while plugged in for charging. This causes "the device
disappears from scans" symptom.

To keep it always advertising:

```bash
cd py
.venv/bin/python wit_write_lowpower.py 3600   # 1 hour
# or 65535 for ~18 hours
```

Verified on 2026-05-25 that the value is persistent across power cycles
(survives full battery disconnect, not just reboot).

## Notes

- The WT9011DCL has a USB‑C port but on many revisions the connector is
  charge‑only — data is over BLE. Same characteristic UUIDs apply.
- BlueZ ≥ 5.66, Linux. No pairing required for notifications.
- The chip is **register-locked by default** — any config write attempt
  without an unlock first will silently fail. The unlock is per-session
  (you may need to repeat it after disconnect/reconnect, untested).
