# test-auto-wit

Minimal Python and Rust BLE clients for a **WitMotion WT901BLE** 9‑axis IMU.

> Tested device: **WT901BLE67** (BLE 5.0). The same protocol is used by other
> WitMotion BLE IMUs (e.g. WT9011DCL) — see notes below.
>
> ⚠️ All code in this repository was generated by an AI assistant
> (GitHub Copilot / Claude). Treat it as a starting point; review before use
> in anything important.

## What it does

Connects over BLE to the sensor, subscribes to the notify characteristic
(`0000ffe4-...`), parses the 20‑byte `0x55 0x61` combined frames
(accel + gyro + Euler angles) and prints decoded values.

## Layout

- `py/` – Python implementation using [`bleak`](https://github.com/hbldh/bleak)
- `rs/` – Rust implementation using [`btleplug`](https://github.com/deviceplug/btleplug)

## Python

```bash
cd py
python3 -m venv .venv
.venv/bin/pip install bleak
.venv/bin/python wit_ble.py [MAC] [N_SAMPLES]
```

Default MAC is hard‑coded in the script; override on the command line.

## Rust

```bash
cd rs
cargo run --release
```

The MAC is currently a constant in `src/main.rs`.

## Frame format (0x55 0x61)

20 bytes, little‑endian `i16`s:

| Bytes | Field | Scale |
|------:|-------|-------|
| 0..1  | header `0x55 0x61` | – |
| 2..7  | ax, ay, az | `/32768 * 16` g |
| 8..13 | wx, wy, wz | `/32768 * 2000` °/s |
|14..19 | roll, pitch, yaw | `/32768 * 180` ° |

## Notes

- The WT9011DCL has a USB‑C port but on many revisions the connector is
  charge‑only — data is over BLE. Same characteristic UUIDs apply.
- BlueZ ≥ 5.66, Linux. No pairing required for notifications.
